# PokéLearnOS — Hermes + Local Qwen + Replit Autonomous Build Goal

## Purpose

This file is the **Replit-optimized Hermes Agent goal** for building **PokéLearnOS** with a local Qwen 35B-class model as the primary AI implementation engine.

Hermes acts as the CEO, orchestrator, memory keeper, task router, verifier, and repair-loop manager. Replit is used as the cloud development workspace, preview environment, task runner, and repository host. The local Qwen model is used for reasoning and code generation through an OpenAI-compatible endpoint, Ollama endpoint, vLLM endpoint, LM Studio endpoint, or another Hermes-supported local provider.

This document is designed to be saved as `GOAL.md` inside the Replit project and used as the top-level autonomous mission for Hermes.

The target behavior is:

> Hermes must not stop after planning. Hermes must build the repository, write the app, write tests, write documentation, generate Replit configuration, create a reproducible ISO/export pipeline, run verification, repair failures, and drive the project as close to 100% completion as Replit safely allows.

# 2. Replit Operating Model

Replit is used for:

- repository generation
- source code editing
- app preview
- backend API testing
- frontend smoke testing
- SQLite initialization
- asset manifest generation
- unit and integration tests
- documentation generation
- Replit-native workflows
- reproducible build orchestration

Replit is **not** treated as the final OS imaging environment unless the workspace supports the required privileges. ISO generation may be implemented as scripts that run locally, in Docker, in a VM, or in a GitHub Actions runner.

The project must therefore produce two execution paths:

1. **Replit Development Path** — run and test the app in Replit preview.
2. **Kiosk/ISO Release Path** — export the repository and run the ISO build scripts in an environment that supports privileged image creation.

Do not let the ISO path block application development in Replit.

- **Backend:** Python 3.12 + FastAPI

# 7. Local Qwen LLM Integration

The application must not depend on cloud LLMs.

Primary LLM target:

```text
Qwen3.6-35B-A3B or closest Qwen 35B-class local model
```

Provider endpoint examples:

```env
LLM_PROVIDER=qwen_local
QWEN_BASE_URL=http://host.docker.internal:8000/v1
QWEN_MODEL=qwen3.6-35b
QWEN_API_KEY=not-required-for-local
```

Alternative endpoint examples:

```env
LLM_PROVIDER=ollama
OLLAMA_HOST=http://your-local-host:11434
OLLAMA_MODEL=qwen3.6:35b
```

Disabled mode must work:

```env
LLM_PROVIDER=none
```

LLM usage is limited to:

- positive feedback variation
- parent dashboard summaries
- optional learning recommendations
- optional Spanish phrasing support

LLM must not be used for:

- core math correctness
- unrestricted child chat
- API key exposure
- internet-required gameplay
- unsafe or age-inappropriate content

Fallbacks must be deterministic.

Expected command behavior:

| `make export-release` | package a release bundle from Replit |
| `make iso` | run or document the ISO build path; may require local privileged environment |

The Replit Run button should use the same logic as `make dev`.
